# Layered Bank and Wallet Mobile Payment Simulation System (Class 12)

## 1) Complete Working Explanation

This project simulates a realistic layered digital payment model where money **cannot move directly from bank to another user**. The only allowed path is:

**BANK → WALLET → OTHER WALLET**

### Technology Stack
- Frontend: Tkinter GUI
- Backend: Python 3
- Database: MySQL
- Connector: `mysql-connector-python`
- Password Security: SHA-256 hashing
- Unique IDs: `uuid`
- Receipt: `reportlab` PDF
- Notification: polling every 5 seconds (no socket use)

### Main Modules
1. **Authentication module**
   - Register user with name, username, password.
   - Password stored as SHA-256 hash.
   - Auto-creates `wallet_uid`, `bank_uid`, wallet row, bank account row.

2. **Transaction module**
   - Supports `BANK_TO_WALLET`, `WALLET_TO_BANK`, `WALLET_TO_WALLET`.
   - Validates positive amount, max amount ₹99999, and balance checks.
   - Uses SQL row locks (`FOR UPDATE`) and transaction commit/rollback.

3. **Dashboard module**
   - Displays user and wallet balance.
   - Shows last 30 minutes transactions.
   - Includes secure password-protected viewing of bank balance and full history.

4. **Receipt module**
   - Generates `receipt_<txn_id>.pdf` after every successful transaction.

5. **Notification module**
   - Polls database every 5 seconds.
   - Shows popup for incoming wallet-to-wallet credit.

### How To Run
1. Install MySQL and create database schema:
   ```bash
   mysql -u root -p < schema.sql
   ```
2. Install dependencies:
   ```bash
   pip install mysql-connector-python reportlab
   ```
3. Run app:
   ```bash
   python payment_simulation.py
   ```

Environment variables (optional):
- `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`

## 2) Flow Diagram Explanation

### Registration Flow
1. User enters name, username, password in GUI.
2. Password hashed using SHA-256.
3. Wallet UID generated by UUID.
4. Bank UID generated as 10-digit random number with prefix.
5. Insert into `users`.
6. Insert wallet with balance 0.
7. Insert bank account with demo balance 50000.
8. Commit transaction.

### Login Flow
1. User enters username and password.
2. System fetches stored hash.
3. Entered password hash compared.
4. On match, dashboard opens.

### Layered Transfer Flow
- **Bank → Wallet**: deduct from bank, add to wallet, record transaction, generate receipt.
- **Wallet → Bank**: deduct from wallet, add to bank, record transaction, generate receipt.
- **Wallet → Wallet**: deduct sender wallet, add receiver wallet, record transaction, generate receipt.

Each transaction is atomic and validated.

### Notification Flow
1. After login, app stores current time.
2. Every 5 seconds, DB query checks incoming wallet-to-wallet transactions newer than last poll.
3. Popup shown if new transaction exists.
4. Last poll timestamp updated.

## 3) Why Layered System Is Realistic

- In real payment ecosystems, users often preload wallet from a bank account.
- Merchants/users receive wallet credits or settlement-ledger credits.
- Layering helps:
  - risk control,
  - traceability,
  - transaction categorization,
  - controlled exposure of core bank balance.
- This model demonstrates banking separation, security, and ledger clarity suitable for academic demonstration.

## 4) Viva Questions with Answers

1. **Why hash passwords?**  
   So raw passwords are never stored in database; this improves security.

2. **Why use UUID for wallet?**  
   UUID provides globally unique identifiers, reducing collision risk.

3. **Why use SQL transactions?**  
   To ensure atomic updates so debit and credit happen together or not at all.

4. **What is `FOR UPDATE` used for?**  
   It locks selected rows during transaction to avoid race conditions.

5. **Why not bank-to-bank direct transfer in this project?**  
   Project requirement enforces layered payment architecture.

6. **How are notifications implemented without sockets?**  
   By periodic polling every 5 seconds to query new records.

7. **What happens if app crashes during transfer?**  
   Database rollback prevents partial updates.

8. **Why generate PDF receipts?**  
   To provide transaction proof and improve realism.

9. **Can this run offline?**  
   Yes, it only needs local Python and local MySQL.

10. **How to demonstrate multi-user transfers?**  
    Open two app windows, log in as different users, transfer wallet-to-wallet.

## 6) Limitations

- Educational simulator only; not connected to real banking rails.
- Uses polling (not event-driven push), so notification may delay up to 5 seconds.
- Basic role model (single user role).
- Local machine DB dependency.
- Float type used as per project requirement; production systems prefer decimal.
